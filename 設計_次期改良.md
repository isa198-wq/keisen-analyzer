## §1 現状アーキテクチャ（3分で把握する）

```
fetch_data.py     yfinanceで日経225の日足OHLCVを取得（auto_adjust=True 必須）
                  → screening_data.csv（縦持ち: 銘柄"コード:社名",日付,始値,高値,安値,終値,出来高）
build_analysis.mjs src/App.jsx から判定関数群を波括弧マッチングで抽出
                  → src/analysis.generated.mjs（手で編集しない。App.jsx変更後は必ず再生成）
screen_daily.mjs  CSVを読み analyze() で全銘柄判定
                  → signals/signals_日付.html（ダッシュボード）＋ LINE通知（Make.com Webhook経由）
                  → signals/state_日付.json（🆕差分用スナップショット。フェーズ1で廃止予定）
                  → signals/history.jsonl（答え合わせ用のシグナル履歴。日付キー・コードのみ）
evaluate.mjs      履歴の蓄積(upsertEntry/seedHistory)と成績集計(evaluate)。単体実行可
backtest.mjs      三尊/逆三尊完成イベントの過去検証。単体実行可
.github/workflows/daily-screening.yml
                  毎朝07:30 JST: データ取得→再生成→スクリーニング→通知。
                  signals/ は actions/cache（key=run_id, restore-keys=signals-state-）で
                  前回実行から引き継がれる（ローリングキャッシュ）
```

判定ロジックの正は `src/App.jsx` の `analyze()`（`detectPattern()` が三尊/逆三尊）。
`analyze(series,"日")` の戻り値で使うフィールド:
- `verdict/vIdx`（4=強い買い, 0=強い売り）, `score`, `factors:[{k,v,s}]`, `trend`, `last.close`, `last.rsi`
- `pattern`: null または `{kind:"top"|"inverse", status:"forming"|"confirmed", broke, breakI, neckLevel, target, stop, rr, quality, vol:{profile}, p1,p2,p3, neckAt(i)}`

### これまでの検証で分かっている事実（設計判断の根拠）
1. 強い買い/売りは**下落局面の10日間ではエッジゼロ**、シードした60日窓では買い+2.2%/10日の優位。
   → 「上げ相場では効くが急落では守らない」仮説。**シグナルの有効性は地合いで変わり、時間とともに
   変わりうる（非定常）**。これがフェーズ2（適応型評価）の存在理由。
2. **張り付き問題**: 同一銘柄が10日連続「強い買い」のまま-14%（9984）等。状態遷移が見えないのが弱点。
3. 週足◎（週足でも同型）は**信頼度の根拠として不支持**（バックテストでむしろ悪化）。情報表示のみ。
4. 2026-06-18の株式分割4銘柄で auto_adjust=False が事故った。データ健全性チェック（前日比1.4倍）を導入済み。
5. スコアの重み（factors の s 値）は全て手決めで未校正。

## §2 不変条件（壊すと事故るもの・必読）

- **recharts は v2 固定**。v3にすると App.jsx が即クラッシュ。package.json を触らない。
- **App.jsx の判定関数を変更したら必ず `node build_analysis.mjs`**。CI もこの順で実行している。
  本設計のフェーズ1〜5は App.jsx を変更**しない**（触る必要が出たら設計ミスを疑うこと）。
- **fetch_data.py の `auto_adjust=True` を絶対に戻さない**（分割事故の再発）。
- **history.jsonl の個別株シグナルは銘柄コードのみ保存**。個別株の価格を保存しない（リターンは
  常に最新の調整済みCSVから日付で引く。これで後日の株式分割でも成績計算が壊れない）。この原則を
  維持する。例外: `ctx:{vix,…}` のような**市況コンテキストの少数スカラー**は保存してよい（§9）。
- **LINE通知は4900字でクリップ**される。メッセージは重要度順（サマリ→🆕→パターン→買い/売り）。
  行を足したら strong モードで実行し本文長を確認、超えるなら reportLines の cap（現25）を20に下げる。
- **CIの signals/ プルーン**（screen_daily.mjs 内、GITHUB_ACTIONS時のみ）は
  `state_*.json` と `signals_*.html` を直近14件に間引く。**history.jsonl 等の新ファイルを
  誤って消さない**よう、追加ファイルはプルーン正規表現に一致しない名前にする。
- evaluate.mjs の `evaluate()` は history エントリの `buys/sells/topsNew/invsNew` キーを読む。
  **このキー名と意味（topsNew=当日完成のみ）を変えない**。キーの追加は自由。旧スキーマの行を
  読んでも落ちないこと（`e.tops || []` 方式）。
- **適応型の仕組み（フェーズ2）は表示・優先度の調整のみ**。シグナルを黙って握りつぶさない、
  スコアの重みを自動変更しない。
- コミットメッセージは日本語・本文つき（過去ログ参照）。`signals/` と `screening_data.csv` は
  gitignore 済みでコミットしない。

## §3 フェーズ1: シグナル遷移の追跡（張り付き対策）

### 目的
「いつから出ているシグナルか」「出てから当たっているか」「今日消えたシグナルはどれか」を見えるようにする。
ワースト事例（9984が買いのまま-14%）は、継続日数と起点比が見えていれば防げた。

### 仕様
1. **継続日数と起点比**: 買い/売りの各銘柄に「そのシグナルが何営業日連続で出ているか(days)」と
   「シグナル初日の終値→現在の騰落率(sinceRet)」を付与。
   - HTML: 買い/売り表に「継続」列（例 `3日目 +2.1%`。sinceRet は買いなら+緑/-赤、売りは逆）。
   - LINE: 各行の tag に日数だけ追記（例 `(3d)`）。1日目は🆕が既にあるので付けない。
2. **張り付き警告**: `days >= 5 かつ 方向逆行が3%超`（買いで-3%以下 / 売りで+3%以上）の銘柄を
   「⚠ 外れ続けている張り付き」としてHTMLの買い/売り表の直前とLINE（パターンの後・買いリストの前、
   最大5行）に列挙。定数 `STALE_DAYS=5`, `STALE_ADVERSE=0.03` を screen_daily.mjs 冒頭付近に置く。
3. **本日解除リスト**: 昨日買い/売りにあって今日消えた銘柄を「🔚 本日解除」として表示
   （例 `🔚買い解除 7203 トヨタ（5日間 +3.2%）`）。HTMLは🆕ブロック相当の位置、LINEは🆕ブロックの直後。
   最大10行。パターン（三尊/逆三尊）の解除は検出器のフリッカーでノイズになるため**対象外**。
4. **🆕判定と状態スナップショットの一本化**: 現在 `state_日付.json` との比較で🆕を出しているが、
   history.jsonl に全情報を持たせて **state_*.json の読み書きを廃止**する（プルーンの削除処理は
   古いファイル掃除のため残してよい）。
   - history エントリのスキーマを拡張: `tops:[{c,s}]`, `invs:[{c,s}]`（s は "f"=forming / "c"=confirmed）。
     既存の `topsNew/invsNew`（当日完成のみ・evaluate用）は**そのまま残す**。
   - 🆕の意味は現行踏襲: 買い/売り = 前営業日エントリに無い（=days===1）。
     パターン = 前エントリに無い、または前が"f"で今日"c"（昇格）、または brokeBarsAgo===0。
   - `seedHistory()`（evaluate.mjs）も tops/invs 全状態を記録するよう拡張（analyze結果の pattern が
     存在すれば kind と status を記録。screen_daily 側の「完成は直近10日以内」の絞り込みはシードでは不要）。

### 実装ガイド
- 連続日数の算出は evaluate.mjs に `streaks(history, key)` としてエクスポート（history は日付昇順前提）。
  `Map<code, {days, start}>` を返す。最新エントリの銘柄集合から開始し、1日ずつ遡って
  「前日にも居たら days++ と start 更新、居なければ確定」という素直なループで書く。
- sinceRet: evaluate.mjs 内に code→{closes, idx:Map(date→i)} を作る共有関数（evaluate() 内の
  byCode 構築を切り出す）を用意し、`closes.at(-1) / closes[idx.get(start)] - 1` で計算。
- 解除リスト: `history.at(-2)` の buys/sells と今日のセットの差分。日数と成績は
  `streaks(history.slice(0,-1), key)` で「昨日時点の連続日数」を取る。
- **処理順序に注意**: screen_daily.mjs では upsertEntry で今日を書き込んだ**後**に loadHistory して
  streaks を計算する（今日を含む連続日数になる）。解除リストも同じ履歴の `at(-2)` が昨日。
- 週末・祝日は history に営業日しか入らないので「連続」はエントリ列の連続でよい（暦日は見ない）。
  CI実行が1日欠けると連続が切れて全部🆕になるが、旧state方式でも同じだった。許容。

### 受け入れ確認
```
SIGNALS=strong node screen_daily.mjs
# → HTML: 買い/売り表に「継続」列。5日以上逆行銘柄がいれば⚠ブロック。state_*.json が新規生成されない
# → 2回連続実行しても history.jsonl の行数が増えない（日付上書き）
# → LINE想定本文が4900字未満（超えるなら cap を 20 に）
```

## §4 フェーズ2: 適応型の答え合わせ（過去の答えが今後も正しいとは限らない問題への対処）

### 目的
現在の「📈自動答え合わせ」は**全期間の平均**であり、過去に引きずられる。市場は非定常で、
シグナルの有効性は (a) 時間経過で劣化しうるし、(b) 地合い（上昇/下落）で反転しうる
（§1-事実1: 買いシグナルは60日窓で+2.2%だが下落10日間ではゼロ）。
「いま・この地合いで効いているか」を継続測定し、効いていないカテゴリに**自動で警戒表示**を出す。

### 設計原則（先に読む）
- **調整は表示と優先度だけ**。シグナル自体は全件出す（情報を隠すと検証も止まる）。
- **仕組みは全部ダム（透明）に**: 閾値は定数、状態遷移は単純な規則。学習・最適化はしない
  （60〜250日のサンプルで適応的に最適化するとノイズを追いかけて必ず過学習する）。
- **サンプル不足なら「判定保留」**と正直に出す。無理に白黒つけない。

### 仕様
1. **窓別成績（鮮度の可視化）**
   - evaluate.mjs の `evaluate()` にオプション `{lastN}` を追加。**窓は「答えが出た（matured）
     シグナル日」で数える**: ホライズン h の集計では、まず `date のインデックス + h <= 最終日`
     を満たすエントリだけに絞り、その末尾 N 日分を使う。
     ※単純に `history.slice(-20)` にすると h=20 ではほぼ全件が「答え待ち」で n=0 になる。これが罠。
   - レポートの「📈自動答え合わせ」表を拡張: 各カテゴリ×ホライズンで「全期間」と「直近60日窓」を
     並記し、劣化方向を ↗（直近が全期間より+0.5%pt以上良い）/→/↘（同悪い）で表示。
     列が多くなりすぎる場合、直近窓の並記は10日ホライズンだけでもよい（勝率・対市場・n は必須）。
   - 表の注記に**構造的ラグ**を明記: 「直近60日窓」は答えが出たシグナルの直近60日分であり、
     h日後の成績は最短でもh日前のシグナルまでしか反映されない。
2. **レジーム判定（地合いの文脈）**
   - 等ウェイト市場プロキシを構築: 全銘柄の日次平均リターンから累積指数
     `index[t] = index[t-1] × (1 + mean(全銘柄の当日リターン))`、`index[0]=100`。
     日付グリッドは最頻の長さを持つ銘柄群から取る。
   - レジーム分類（その日の値で決定的に）:
     `上昇 = index > SMA25(index) かつ SMA25が5日前より上`、
     `下落 = index < SMA25 かつ SMA25が5日前より下`、それ以外は `もみ合い`。
   - 毎実行CSVから全日付分を導出（保存しない。決定的なので再現できる）。
   - 現在のレジームをHTMLヘッダ（地合いバーの隣）とLINEサマリに表示。
3. **レジーム条件付き成績**
   - `evaluateByRegime(groups, history, regimes)`: カテゴリ×レジーム×ホライズンで全期間集計。
   - レポートに「いまのレジームでの成績」を1行で出す。例:
     `現在は下落レジーム ｜ 下落時の成績(10日後): 買い 勝率38%(対市場-0.4%, n=312) / 売り …`
     これが「過去の答えを今の文脈で読み替える」表示の中核。
4. **信頼度状態（アジャストの本体）**
   - カテゴリごと（buys/sells/topsNew/invsNew）に3状態: `✅有効` / `⚠️警戒` / `❔判定保留`。
   - 入力は2つ（どちらもホライズン10日の対市場edge）:
     A = 直近60日窓のedge、B = 現在レジームにおける全期間edge。
   - 規則（定数は screen_daily 冒頭）:
     - n不足（買売: 窓n<100 または レジームn<100 ／ パターン: n<8）→ `❔判定保留`
     - A と B の両方 > TRUST_EXIT(+0.3%) → `✅有効`
     - A または B < TRUST_ENTER(−0.3%) → `⚠️警戒`
     - どちらでもない（バンド内）→ **前日の状態を維持**（ヒステリシス。日次フリップ防止）
   - 前日状態は history エントリに `trust:{buys,sells,topsNew,invsNew}` として毎日保存し、
     翌日 `history.at(-2).trust` から読む（無ければヒステリシスなしで規則を直接適用）。
   - **効果（表示のみ）**:
     - HTML: 各セクション見出しに状態チップ（例 `🔴 買いサイン ⚠️直近効いていません`。
       ✅は控えめに、⚠️は目立つ色 AMBER/RED 系で）。
     - LINE: サマリの📈行の直後に状態行を1行（例 `信頼度: 買⚠️ 売✅ 三尊❔ 逆三尊❔`）。
     - ⚠️カテゴリの🆕ハイライト行の行頭に ⚠ を付ける（載せるのはやめない）。
     - 通知の自動ミュートは**しない**。将来用に定数 `ADAPTIVE_MUTE = false` の入り口だけ
       置いてよい（実装は不要）。
5. **シード増量（レジーム別サンプルの確保）**
   - 自動シードとCLIシードの既定を 60→**250** 営業日に（2年データ=488本でも idx≥80 を満たす範囲）。
     実行時間は約4〜5分、自動シードは履歴が薄い時しか走らないので許容。
     フェーズ4（5年化）後はさらに厚くできるが必須ではない。
6. 定数（screen_daily.mjs 冒頭にまとめる）:
   `TRUST_ENTER=-0.003, TRUST_EXIT=+0.003, TRUST_MIN_N=100, TRUST_MIN_N_PAT=8, RECENT_WINDOW=60`

### 実装ガイド
- evaluate.mjs に追加してエクスポート:
  - `buildMarketIndex(groups)` → `[{date, value, sma25, slope}]`
  - `classifyRegimes(groups)` → `Map(date → "up"|"down"|"range")`
  - `evaluate(groups, history, opts?)` — 既存呼び出し（引数2つ）はそのまま動くこと
  - `evaluateByRegime(groups, history, regimes)`
  - `computeTrust(prevTrust, windowStats, regimeStats, consts)` → `{buys:"ok"|"warn"|"hold", …}`
- CLI（`node evaluate.mjs`）も窓別＋レジーム別＋現在レジームを表示するよう printReport を拡張。
- SMA25・5日傾きは evaluate.mjs 内に素朴に実装（analysis.generated の sma は series 用なので流用しない）。
- LINEの追加は最大2行（レジーム行は既存サマリ行に統合してよい）。4900字確認を忘れない。

### エッジケース
- 履歴が窓より短い → その窓・その状態は「蓄積中/❔」。
- レジームは等ウェイトプロキシなので日経平均と多少ずれる。それでよい（このツールが見ている
  銘柄群に対する地合いを測るのが目的）。
- seedHistory で補完した過去日には trust が無い → 翌日以降の実行で自然に埋まっていく。
- topsNew/invsNew はイベント数が少なく、当面ほぼ `❔判定保留` になる。それが正直な表示。

### 受け入れ確認
```
node evaluate.mjs            # 窓別＋レジーム別の表、現在レジームが表示される
SIGNALS=strong node screen_daily.mjs   # 2回連続実行
# → HTML: 現在レジーム表示・↗/→/↘つき成績表・セクション見出しに状態チップ
# → history.jsonl 最終行に trust キーがあり、2回目実行で状態がフリップしていない
# → LINE想定本文 4900字未満
```

## §5 フェーズ3: golden tests（回帰の保険）

### 目的
build_analysis.mjs の波括弧マッチング抽出は App.jsx の書き方変更で静かに壊れうる。
壊れたまま通知が飛ぶのを CI で止める。

### 仕様
`run_tests.mjs`（新規・フレームワーク不使用、失敗時 `process.exit(1)`）:
1. **抽出健全性**: `src/analysis.generated.mjs` から `analyze, buildSeries, tfSeries, detectPattern` を
   import し、全て typeof "function" であること。
2. **golden 逆三尊**: 決定的な合成OHLCV（乱数不使用・数値ハードコード）で
   `detectPattern(buildSeries(fixture))` が `{kind:"inverse", broke:true}` を返すこと。
   フィクスチャ設計の目安: 80本以上。下降→左肩安値(例900)→戻り(960)→頭(850)→戻り(970)→
   右肩(905)→ネック上抜けラリー(1000超)。V字は前後4本より深く（findSwings w=4）、
   肩差<8%・深さ>1.5%・肩間スパン12〜110本の検出条件を満たすこと。
   open=前日close、high=max+2、low=min-2、volume=1e6（ブレイク日だけ2e6）。
   一発で検出されなければ `console.log(JSON.stringify(p))` でデバッグして数値を調整し、
   最終的に**調整済みの数値をハードコード**する。
3. **analyze スモーク**: 同フィクスチャで `analyze(series,"日")` が
   `vIdx: 0..4 の整数`・`factors: 配列`・`factors に k==="逆三尊" を含む` こと。
4. **CSVパーサ**: screen_daily と同じ分割ロジックを5行のインライン文字列で検証
   （ヘッダスキップ・銘柄列の"コード:社名"分解・日付昇順）。
5. **（フェーズ2導入後）** `classifyRegimes` のスモーク: 単調上昇の合成データで "up" が返ること。

### CI 組み込み
daily-screening.yml の「判定ロジックを App.jsx から再生成」ステップの直後に:
```yaml
      - name: テスト（判定ロジックの健全性）
        run: node run_tests.mjs
```
テスト失敗＝ジョブ失敗＝通知が飛ばない（壊れたシグナルを配らない）。これは意図した挙動。

### 受け入れ確認
`node run_tests.mjs` がローカルで exit 0・全項目 PASS 表示。わざと analysis.generated.mjs の
export を1つ消すと exit 1 になること（確認後戻す）。

## §6 フェーズ4: 履歴5年化＋バックテスト範囲修正

### 目的
現在2年分（日足488本）でパターン完成イベントが58件しかなく、統計が弱い。
また **backtest.mjs には範囲バグがある**: `buildSeries(tfSeries(bars,"D"))` は直近160本に
キャップされるため、PERIOD を伸ばしてもバックテスト範囲が広がらない。

### 仕様
1. fetch_data.py: `PERIOD = "5y"` に変更（コメントも更新）。
2. backtest.mjs: 成長窓を**生の bars ベース**に変更（evaluate.mjs の seedHistory と同じ方式）:
   ```js
   // 変更前: const full = buildSeries(tfSeries(bars, "D")); … full.slice(0, t+1)
   // 変更後: 各 t について buildSeries(tfSeries(bars.slice(0, t+1), "D")) を判定に使い、
   //          フォワードリターンは bars[t+h].close / bars[t].close で生バーから引く
   ```
   ループ範囲は `t = MINBARS .. bars.length-1-MAXH`。実行時間が数分になるため
   進捗表示（10%刻み）を入れる。週足◎分割検証の rawOffset 計算は生バー基準になり単純化できる。
3. evaluate.mjs: seedHistory の `bars.length !== L` スキップは「最初の銘柄と同じ長さ」前提。
   5年化で本数バラつき（上場が浅い銘柄）が出るので、**最頻の長さと比較する**よう直す。
4. データ再取得と確認: `python fetch_data.py` 後、前日比1.4倍超ジャンプが0件であること
   （screen_daily 実行時の🚨警告で確認）。
5. history.jsonl の既存エントリはそのまま有効（日付で引くだけなので互換）。CSV窓から外れた
   古いシグナルは evaluate の idx==null スキップで自然に評価対象外になる（仕様どおり）。

### 受け入れ確認
```
python fetch_data.py            # 完了行の行数が2年時(約10.9万行)の2倍超
node backtest.mjs               # シグナル総数が現状(58)から大幅増（数百件目安）
node run_tests.mjs              # フェーズ3のテストが引き続き通る
SIGNALS=strong node screen_daily.mjs   # 🚨データ異常警告が0件
```

## §7 フェーズ5: ファクター別成績の計測（重み校正の下準備）

### 目的
analyze() の factors（ゴールデンクロス、RSI、三尊…）の s 値（重み）は全部手決め。
どのファクターが当たっていて、どれがノイズ/逆指標かを**測る道具**を作る。
**このフェーズでは重みを変更しない**（変更はデータが半年貯まってから別途判断）。

### 仕様
`factor_check.mjs`（新規・単体実行専用。CIには入れない）:
- 全銘柄×過去N営業日（既定120、`--days N`で変更可）を成長窓でリプレイ
  （`buildSeries(tfSeries(bars.slice(0,i+1),"D"))` → analyze。フェーズ4のbacktest方式と同じ）。
- 各日の `factors` を走査し、ファクターごとに
  `alignedRet(h) = sign(s) × (bars[i+h].close/bars[i].close - 1)` を 5/10/20日で収集。
  同日・同ホライズンの全銘柄平均（ベースライン）も収集し、
  `edge = alignedRet - sign(s)×baseline` を計算（方向調整済み優位性）。
- **フェーズ2の classifyRegimes を再利用し、レジーム別の内訳も集計**する
  （「このファクターは上昇時のみ有効」を発見できるように）。
- 出力: ファクター名ごとに n / 出現率 / 各ホライズンの edge / 勝率を edge(10日)降順で表示。
  末尾に「edge がマイナスのファクター＝見直し候補」を列挙。n<30 は「参考」欄に分離。
- 結果を `signals/factor_stats.json` に保存（プルーン正規表現に一致しない名前）。
- 実行時間目安: 数分。進捗表示を入れる。

### 実装メモ
- ファクター名 k は App.jsx の analyze() 内の `factors.push({k:"…"})` を grep して確認し、
  実際の文字列をそのまま集計キーに使う。s===0 は集計から除外。

### 受け入れ確認
`node factor_check.mjs --days 120` が数分で完走し、テーブル表示＋factor_stats.json 生成。
クロス系・RSI系ファクターの n が数千オーダーであること。

## §9 フェーズ7: 市況コンテキストの拡張（VIX恐怖指数・ビットコイン）【独立・小規模・前倒し可】

### 目的
個別株シグナルの外側にあるリスクセンチメント——米国の恐怖指数（VIX）とリスク選好の代理指標（BTC）——を
毎朝のレポートとアプリで見えるようにする。VIX急騰局面は「買いシグナルが効かない地合い」（§1-事実1）の
有力な外部指標候補。まず**表示と記録**から始め、判定には使わない（§10）。

### 前提知識
- 市況データは `fetch_market.py` → `public/market_data.csv`（screening_data.csv と同じ縦持ち形式）。
  CI・「毎日スクリーニング.bat」とも screen_daily の前に実行される。
- **アプリの市況バーは CSV の内容を動的に全表示する**（App.jsx 1468行付近: parseLongFormat →
  market:"IDX"。表示名も CSV の銘柄列をそのまま使う）。従って **App.jsx の変更は不要**。

### 仕様
1. **fetch_market.py**: INDICES に2行追加し、PERIOD を "6mo"→"2y" に変更（下記4のctxバックフィル用。
   7指標×約500行なので軽い）:
   ```python
   ("^VIX", "VIX恐怖指数"),
   ("BTC-USD", "ビットコイン"),
   ```
   - auto_adjust は指数・為替・BTCには実質無関係なので現状（False）のままでよい。
   - VIX の出来高は0になるが既存コードで対応済み。BTC-USD は土日も日足があるが縦持ち形式なので問題ない。
   - BTC-JPY ではなく BTC-USD を使う（データの安定性優先。ドル円が隣に表示されるので換算は容易）。
2. **アプリ**: 変更不要。`npm run dev` で市況バーに2指標が増えることを目視確認するのみ。
3. **日次レポート（screen_daily.mjs）に🌍市況ストリップ**:
   - `public/market_data.csv` を読む。**無ければセクションごとスキップ**して正常動作すること
     （ローカルで fetch_market.py を走らせていないケースがあるため）。
   - 指標ごとに小カード（指標名・終値・前日比%・直近30日スパークライン=既存 sparkline() 再利用）を
     ヘッダ直下（stats グリッドの上）に横並び表示。CSVにある指標を全部出す（動的）。
   - **VIXの特別扱い**: 水準バッジ `<20 平穏(緑) / 20〜30 警戒(琥珀) / ≥30 恐怖(赤)`。
     前日比の色は**反転**（VIX上昇=リスク悪化=赤）。VIXは変化率より水準に意味があるため。
   - 鮮度: 最終日付が5暦日より古い指標は値の横に `(MM/DD)` を付けて明示（休場・取得失敗の可視化）。
   - LINE: サマリの後に1行だけ追加（長さ節約のため4つに限定）:
     `🌍 日経-0.8% ドル円157.2 VIX22.4⚠ BTC$67.4k+2.1%`（4900字確認を忘れない）。
4. **history.jsonl に市況コンテキストを記録**（将来の条件付き検証用。**今回は記録のみ**）:
   - 日次エントリに `ctx:{vix, nk1d, btc7d}` を追加（vix=終値水準、nk1d=日経225の前日比%、
     btc7d=BTCの7日騰落%。取れない値はキー省略）。
   - **as-of結合**: シグナル日付 d に対し「d以下で最新の日付」の値を使う（VIXは米国営業日、
     BTCは毎日、と暦が違うため）。
   - seedHistory も market_data.csv があれば過去分の ctx をバックフィルする（PERIOD=2y はこのため）。
   - 将来拡張（今回はやらない）: 「VIX>25 の日の買いシグナル成績」のような ctx 条件付き集計を
     §4 の窓/レジーム集計に追加する。データが貯まってから別途設計。

## §10 非スコープ（やらないと決めたこと）

- **スコア重みの自動チューニング／適応的な閾値の自動最適化**: サンプル不足で過学習必至。
  適応型の仕組み（§4）はあくまで「窓とレジームで成績を読み替えて表示する」まで。
- **⚠️カテゴリの通知自動ミュート**: 情報を隠すと検証データも途切れる。当面は警戒表示のみ
  （`ADAPTIVE_MUTE` 定数の入り口だけ許可、実装しない）。
- **週足◎を信頼度・優先度に使うこと**: バックテストで不支持（§1-3）。表示のみ維持。
- **VIX・BTCをシグナル判定やスコアに直接入れること**: まず表示と ctx 記録のみ（§9）。
  条件付き検証でデータ的裏付けが出たら別途設計する。
- **新しいテクニカル指標の追加**（一目均衡表等）: 既存ファクターの検証が先。
- **リアルタイム/分足対応・UIのさらなる装飾**: 日足スイングの道具として不要。
- App.jsx（ブラウザアプリ側）への機能追加: 本設計は全てヘッドレス側で完結する。
