## §1 目的と背景

### 前提となる確定事項（2026-07-06・ユーザー確定、監査移植レポート冒頭）

1. **価格・出来高のみのシグナル研究はクローズ**（v2〜v4＋イナゴ盤監査、5回連続採用ゼロ）。
2. イナゴ盤は「シグナル生成器」ではなく**状況把握ダッシュボード**として位置づける。
3. 将来イナゴのエッジを再検証する場合は**「小型株ユニバース＋情報レイヤー込み」の形に限る**。

**本書はこの決定を維持する。** v6はシグナルを作らない。テーマ探索（点火クラスタ）を
「観察装置として信頼できる品質」に引き上げ、**機能しているか自体を計測可能にする**のが目的。

### 現状（2026-07-10時点・コミット 8970257）

- `inago/detect_clusters.mjs` が日次クラウドジョブ（07:30 JST）で点火クラスタを検知し、
  `screen_daily.mjs` がHTMLレポートとLINE通知に「未検証の仮説」として最大5件表示するところ
  まで完成している（コミット ca5122f）。
- Grok命名（`inago/grok_name.py`）はxAI課金・鍵管理が必要なため**ローカル専用**。
  `run_inago.ps1` はタスクスケジューラ未登録のため、**命名は実質一度も運用されていない**。

### 解決したい問題（重要度順）

1. **【混入バグ】クラスタ母集団に米国株が混ざった**: コミット 8970257（2026-07-10）で
   `screening_data.csv` がナスダック100込みの323銘柄になった。`build_data_js.mjs` はCSV全銘柄を
   `data.js` に変換し、`detect_clusters.mjs` に市場フィルタは無い。東証と米国市場は取引時間が
   ずれており（東証の当日は米国の前日終値を織り込む）、日次リターンの20日相関を市場をまたいで
   計算すると**時差由来の疑似相関クラスタ**が出る。これは設計変更ではなくバグとして即修正する。
2. **クラスタの中身が市場ベータで汚染されている疑い**: 大型223銘柄の素のリターン相関
   （窓20日・閾値0.55）は、市場全体が動いた日の共通変動で底上げされる。実測（2026-07-08、
   ローカル `inago/clusters.json`）のクラスタ例「帝人・エプソン・大和証券G・住友化学・日本精工・
   神戸製鋼」は異業種の寄せ集めで、「同じテーマ」よりも「同じ日に地合いで上げた」の可能性が高い。
3. **答え合わせがない**: クラスタを記録していないため「その後どうなったか」「持続したか」が
   分からず、この機能が観察装置として意味があるのかすら判定できない。本体シグナルには
   `history.jsonl`＋自動答え合わせがあるのに、クラスタだけ計測の外にいる。
4. **名前がない**: LINEに銘柄名の羅列だけが届く。「これは何のテーマか」という一番知りたい情報が
   欠けている。v5フェーズIで蓄積を始めた適時開示（`signals/disclosures.jsonl`）と突き合わせれば、
   **Grokなし・追加鍵なしで**「共通材料の裏付けがあるか」まで言える。

### 評価の哲学（従来どおり）

基準は事前固定・結果を見てから変えない・「クラスタ検知はノイズだった」も正当な結果。
v6の各フェーズは**表示と記録のみ**を変え、売買シグナル・スコア重みには一切触れない。

## §3 フェーズL: 市場分離＋クラスタ履歴・答え合わせ基盤

### 目的
米国株混入を止め、クラスタを日次で記録し始める。**記録が先、評価は蓄積後**（v5フェーズIと同じ
「前向き蓄積の開始点」思想）。

### L-1. 市場分離（`inago/detect_clusters.mjs`）
- クラスタ検知の母集団を**東証銘柄のみ**（コード先頭が数字）にフィルタする。米国株は
  点火判定にもクラスタ相関にも入れない。フィルタは `stocks` 構築直後に1行で入れる。
- 理由をコメントに残す: 時差のある市場間の日次リターン相関は疑似相関になる／イナゴ
  （テーマ物色）は東証の文脈であり、将来方向も国内小型株（§6）。
- `data.js` 自体は全銘柄のまま変えない（`inago_offline.html` の手動ダッシュボードが米国株を
  表示できること自体は害がない。ただしストップ高近似など東証前提のロジックがあるため、
  offline版の検証タブを米国株込みで読まないよう §7 に注意を記録する）。
- 出力ログに「東証XXX銘柄を対象（米国株YY銘柄は除外）」を出す。

### L-2. クラスタ履歴の記録（`screen_daily.mjs`）
- `screen_daily.mjs` が `inago/clusters.json` を読んだ直後に、`signals/cluster_history.jsonl` へ
  1行追記する: `{date: 最新営業日(YYYY-MM-DD), clusters: [{codes:[...]}...]}`。
  - **銘柄コードのみ保存、価格は保存しない**（history.jsonl と同じ不変条件。リターンは常に
    最新CSVから日付で引く）。名前も保存しない（コード→名前は表示時にCSVから引ける）。
  - 冪等: 同一 `date` の行が既にあれば上書き（upsert）。ジョブ再実行で二重計上しない。
  - クラスタゼロの日も `{date, clusters:[]}` を記録する（「出なかった」も答え合わせの分母）。
- ファイル名はプルーン正規表現に非該当（`cluster_history.jsonl` はOK）。

### L-3. 答え合わせの集計と表示（`screen_daily.mjs`、新規 `cluster_eval.mjs`）
- 新規 `cluster_eval.mjs`（単体実行可・screen_daily から import）:
  - 各記録日Dの各クラスタについて、メンバーのD→D+N終値リターン平均（N=5,10）と、
    同期間の**東証ユニバース平均**リターンを最新CSVから計算し、差分（対市場）を出す。
  - **持続日数**: 記録日Dのクラスタと翌営業日D+1の全クラスタとの**メンバーJaccard係数が
    0.5以上**なら「同一クラスタが持続」とみなし、連続日数を数える。
  - 答えが出た（D+Nが存在する）クラスタのみ集計。nと平均対市場、平均持続日数を返す。
- 表示: **HTMLレポートのクラスタセクション末尾に1行だけ**追加
  （例: `クラスタのその後: n=12 / 10日後対市場+0.4% / 平均持続1.8日（蓄積中・参考値）`）。
  **LINEには足さない**（4900字予算を消費しない。クラスタは既に「未検証」明記済み）。
- これは**観察装置の自己診断**であり売買シグナルの成績ではない。翌寄執行・コスト控除などの
  執行モデル（P-1/P-4）は適用しない。「終値ベースの参考値」と表示文言に明記する。

### 事前固定の判断基準（§6 フェーズOで使う。途中で変えない）
蓄積30営業日後に評価する:
- (a) クラスタメンバーの10日後リターンが東証ユニバース平均を上回っているか（対市場>0）。
- (b) 平均持続日数が2営業日以上か（1日で消えるクラスタはノイズの疑い）。

### 受け入れ確認
1. `node inago/build_data_js.mjs && cd inago && node detect_clusters.mjs data.js` で
   米国株がログ・clusters.json に一切現れない。
2. `node screen_daily.mjs` を2回連続実行しても `cluster_history.jsonl` の同一日付行が1行のまま。
3. クラスタゼロの日をシミュレート（clusters.json を `[]` に）しても記録・表示が壊れない。
4. `node run_tests.mjs` 全PASS。LINE本文長がstrongモードで4900字以内（クラスタ行は増えていないこと）。

## §4 フェーズM: ベータ除去（クラスタ品質の改善）

### 目的
「同じ日に市場が上げた」だけの疑似クラスタを減らし、クラスタ＝共通テーマの確度を上げる。

### 変更（`inago/detect_clusters.mjs`）
- `retSeries()` が返す日次リターンを**超過リターン**にする:
  `各銘柄のリターン − 同日の東証ユニバース平均リターン`。
  - ユニバース平均は data.js 内の東証銘柄（L-1フィルタ後）から日付キーで計算する。
    N225指数を外部から読まない（data.js内で完結・欠損に強い・追加依存ゼロ）。
  - 日付整合は**日付キーの突き合わせ**で行う（配列位置合わせは休場ズレで壊れる。
    v2の教訓＝P-5と同種の罠）。
- 相関閾値 `TH=0.55`・窓 `WIN=20`・点火判定は**据え置き**。1コミット1変更の原則で、
  このフェーズの差分は「素のリターン→超過リターン」のみにする（帰属を明確にするため）。
- `inago_offline.html` 側の同名ロジックとの整合: detect_clusters.mjs は冒頭コメントで
  「HTMLと同一ロジック」を謳っている。**HTML側にも同じ超過リターン化を入れるか、入れない
  なら冒頭コメントの「同一」宣言を「クラスタ相関のみ超過リターン版」に更新する**。
  推奨は後者（offline版は手動観察用で、二重メンテのコストの方が高い）。

### 効果の見方（受け入れ確認を兼ねる）
1. 変更前後で同じ data.js に対して clusters.json を出し、構成を並べて目視比較する
   （比較用の使い捨てスクリプトは不要。git stash か一時コピーで前後を出せばよい）。
   期待: 異業種寄せ集めクラスタが減る／クラスタ総数が減る（**ゼロの日が増えるのは正常**、
   それが狙い。「毎日何か出る」方が疑わしい）。
2. 変更後の持続日数・対市場（フェーズLの指標）を以後の蓄積で追う。
3. `node run_tests.mjs` 全PASS（detect_clusters はテスト対象外だが本体を壊していないことの確認）。

## §5 フェーズN: クラスタ命名＋開示突き合わせ（Claude・クラウド）

### 目的
「銘柄名の羅列」を「テーマ名＋材料の裏付け有無」に変える。**X検索は使わない**（Grokはローカル
任意のまま）。代わりにv5で蓄積中の適時開示という**一次情報**で裏付けを取る。幻覚対策を最優先。

### N-1. 命名: 新規 `name_clusters.mjs`（リポジトリ直下・Node・`@anthropic-ai/sdk`）
- 入力: `inago/clusters.json` ＋ `signals/disclosures.jsonl`（直近5営業日・メンバー銘柄分のみ）。
- 各クラスタについて Claude（`claude-haiku-4-5`・構造化出力json_schema）に渡すもの:
  - メンバーの銘柄名・コード。
  - メンバーの直近開示（あれば）: `title` と `cls.kind`・`cls.oneLine`。
- 出力スキーマ（固定）:
  - `name`: string（テーマ名・簡潔な日本語。不明瞭なら「共通テーマ不明瞭」）
  - `catalyst`: string（1文。開示に基づくならその内容、なければ一般知識ベースと明記）
  - `confidence`: enum `高|中|低`
  - `evidence`: enum `開示あり|開示なし`（メンバーの開示で説明できたか）
- **幻覚対策をプロンプトで強制**: 「与えた開示と一般知識で説明できない場合は、無理に
  結びつけず name を『共通テーマ不明瞭』・confidence を『低』とせよ」「存在しない材料を
  作らない」。低confidenceを許容する設計にする（v5の分類と同じ思想: 正直な不明 > 幻覚）。
- 出力: `inago/cluster_names.json`（新規・gitignoreに追加）。キーは構成コード昇順連結
  （`grok_name.py` と同じ規約）。API失敗・キー未設定時は**黙ってスキップしファイルを
  書かない**（`disclosure_classify.mjs` と同じ方式。後段は名前なしで動く）。
- コスト: 1日≤8クラスタ×haiku（入力は数百トークン/件）＝1日1円未満のオーダー。
  `設計_次期改良v5.md` §2 のモデル選定方針に従い、Opus/Fableは使わない。

### N-2. 表示統合（`screen_daily.mjs`）
- `cluster_names.json` があれば読み、クラスタ表示を
  `[A] テーマ名（確度・開示裏付け有無）: 銘柄名…` 形式にする。名前が無いクラスタは現行表示のまま。
- LINE: `LINE_CAP_CLUSTERS = 5` 据え置き。テーマ名追加分の文字数を実測し、strongモードで
  4900字を超えるなら**テーマ名を12字で切り詰める**（capは下げない）。
- HTML: テーマ名に加え `catalyst` も表示。「AI推定・未検証」の注記は現行文言を維持する。

### N-3. ワークフロー統合（`.github/workflows/daily-screening.yml`）
- クラスタ検知ステップの直後・スクリーニングの前に `node name_clusters.mjs` を追加。
  `continue-on-error: true`・`env: ANTHROPIC_API_KEY`（v5と同じSecretを共用。**新規Secret不要**）。

### 受け入れ確認
1. ローカルで `ANTHROPIC_API_KEY` を設定し実データで実行、cluster_names.json の全クラスタに
   スキーマ通りの値が入る。開示ゼロのクラスタで `evidence=開示なし` になることを確認。
2. キー未設定で実行→スキップされ、screen_daily が名前なし表示で正常完了する。
3. LINE想定本文長がstrongモードで4900字以内（実測値をコミットメッセージに書く）。
4. 意図的に壊れたJSONを cluster_names.json に置いても screen_daily が落ちない（try/catch）。

- **(a)(b)のいずれかがYes、かつ命名で「テーマとして解釈できる」クラスタが体感で半分以上**

  **小型株ユニバース（例: 東証グロース市場250構成銘柄）への拡張**を検討する。

  → 大型株でのクラスタ検知はノイズと結論し、通知セクションを撤去または大幅縮小する。
  **撤去も正当な結果**（5回の採用ゼロと同じ扱い。作った機能を消す判断を先送りしない）。

どちらの場合も判断と根拠数値を本書のこの節に追記し、コミットする。

## §7 リスク・制約（必須記載）

1. **鮮度は最大1営業日遅れ**: クラウドジョブは07:30 JST実行のため、クラスタは「前営業日まで」の
   データで検知される。点火当日の場中には気づけない。仕様であり欠陥ではない（場中対応は
   `run_inago.ps1`＋ローカルの守備範囲）。
2. **disclosures.jsonl は前向き蓄積のみ・Yanoshin API停止リスクあり**（v5§2）。開示が空でも
   フェーズNは動く（`evidence=開示なし` になるだけ）。
3. **命名は非決定的**: 同じクラスタでも日によって違う名前が付きうる。confidence と
   「AI推定・未検証」併記で対処し、名前の履歴管理はしない（過剰設計を避ける）。
4. **signals/ キャッシュ消失リスク**: GitHub Actionsキャッシュは未アクセス7日で消える。
   `cluster_history.jsonl` は毎朝アクセスされるため通常は持続するが、ジョブが7日以上止まると
   消える。artifact（signals-report）に毎回含まれるため手動復旧可能。history.jsonl と同等の
   リスクレベルで、追加対策はしない。
5. **inago_offline.html の検証タブは東証前提**（ストップ高近似・地合い判定）。data.js に
   米国株が入った状態で検証タブの数字を読まないこと（フェーズMの整合コメントにも記載する）。
6. クラスタの答え合わせ（L-3）は終値ベース・執行モデルなしの**参考値**。これを根拠に
   売買判断をしない。シグナル復活の判断は §6 の手順以外で行わない。

### 9-4. L-2の修正: 「検知失敗」と「クラスタゼロ」を区別する（重要）

`inago/clusters.json` はgitignore対象でCIの毎回のチェックアウトには存在せず、検知ステップは
`continue-on-error: true`。つまり**検知ステップが失敗した日はファイル自体が無い**。§3 L-2の
「クラスタゼロの日も `{date, clusters:[]}` を記録」をそのまま実装すると、検知失敗日が
「出なかった日」として記録され、フェーズOの事前固定基準の分母が汚染される。読み替え:

- `clusters.json` が**存在し正常にパースできた場合のみ** `cluster_history.jsonl` に記録する
  （`detect_clusters.mjs` はクラスタ0件でも必ず `[]` を書き出すので、ファイルの存在＝検知実行済み）。
- ファイル不存在・パース失敗の日は**何も記録しない**（欠測として扱う。ゼロとして数えない）。
- 記録位置: 「clusters.json を読んだ直後」とあるが、記録に使う日付 `dataDate`（CSVの最終営業日）は
  現行コードでは読込より後で計算される。**`dataDate` 確定後に記録**すればよい（「直後」は不要）。

### 9-5. フェーズMの補足: `norm()` が日付を捨てている（重要）

data.js の各ローソクには `date` があるが、`detect_clusters.mjs` の `norm()` はOHLCVのみ残して
**日付を捨てる**。§4の「日付キーの突き合わせ」を実装するには:

1. `norm()` を拡張して `date` を保持する（`date: c.date` を足すだけ。scoreAt等は影響なし）。
2. 各銘柄について**日付キーのリターンMap**（date → close(d)/close(前営業日)−1）を先に作る。
3. 東証ユニバース平均は「その日付のリターンを持つ銘柄」の平均として日付ごとに計算する。
4. 超過リターン = 銘柄の日付キーリターン − 同日付のユニバース平均。`retSeries()` はこの
   超過リターン系列の直近WIN件を返す形に変える。

`pearson()` の末尾N件位置合わせは現行のまま変えない（銘柄間の欠損日ズレは現行コードにも
ある既知の限界で、本フェーズのスコープ外。1コミット1変更の原則）。

### 9-6. L-3の細目（軽微・実装時の解釈揺れ防止）
