## §1 目的と背景（なぜ「価格の外」へ出るか）

v2〜v4で確定した事実: **このユニバース（日経225の大型223銘柄・5年）では、価格と出来高だけから
作る機械的シグナルに実用的なエッジは無い。** 三尊/オシレーター/トレンド複合（v2）、指数レベルの
退出規則（v3）、モメンタム/出来高フォロー/速い退出（v4）——すべて事前固定基準を満たさず採用ゼロ。

残っている未検証の入力空間は**価格以外の情報**（決算・業績修正・自社株買い・M&A・IR・ニュース）。
ここを人間の速度と量を超えて毎朝処理できるのがフロンティアAIの新しい能力であり、本書はその
「情報レイヤー」を既存パイプラインに接ぐ。

**評価の哲学は従来どおり**: 候補は手で列挙・基準は事前固定・結果を見てから変えない・
採用ゼロも正当な結果。ただし本書には従来にない2つの新しい難しさがある（§6で詳述）:
1. **開示は公開情報**なので、出た瞬間に価格へ織り込まれる。執行遅延の影響がv4以上に致命的。
2. **無料の外部データに依存**し、**LLM分類は非決定的でコストと幻覚を伴う**。

### テスト可能な仮説（本書が狙う唯一の検証対象）

**PEAD（Post-Earnings-Announcement Drift・決算後ドリフト）**: 決算サプライズが出た方向へ、
発表後も数日〜数週間ドリフトする傾向。文献上もっとも頑健とされるアノマリーの一つで、
**本質が「情報（サプライズ）」であって価格ではない**点がv2〜v4と決定的に違う。AIは開示文を
「上方/下方サプライズ」に分類できる。これを既存のバックテスト土台で採点する（フェーズJ）。

### テストしない（が価値がある）フレーミング

「明日上がる銘柄を当てる」ではなく、**「今日なぜ動いたか・何を開示したか・価格シグナルは
実イベントに裏打ちされているか」を毎朝整理する観察装置**。v2§7で定めた本ツールの位置づけ
（予測ではなく整理・観察・記録）そのものであり、PEADが採用に至らなくても表示価値は残りうる
（ただし表示はフェーズKで、Jの結果を見てから判断する）。

## §2 使う部品・データ・外部API

### 既存部品（車輪の再発明禁止）
- `screen_daily.mjs`（毎朝の headless ジョブ・Discord/Slack Webhook）、`history.jsonl`、
  `backtest.mjs`、`evaluate.mjs`（`buildByCode`/`buildDateGrid`/`classifyRegimes`）。
- データ: `screening_data.csv`（5年・223銘柄）。フォワードリターンは常に最新CSVから日付で引く。

### 外部データ: 適時開示（TDnet）
- **主データ源: Yanoshin 東証TDnet WEB-API（無料・JSON・無認証）** — `https://webapi.yanoshin.jp/tdnet/`
  - 銘柄コード指定・日付範囲指定・キーワード検索が可能。数分ごとに東証と同期。
  - 出力: JSON/XML/RSS/Atom/HTML。本書はJSONを使う。
  - **正確なエンドポイント形（クエリパラメータ）は実装時に上記ドキュメントで必ず確認する。**
    本書はエンドポイントURLを決め打ちしない（個人運営APIで仕様変更・停止がありうるため）。
  - **重大な制約**: 個人運営で「突然停止しうる」と明記されている。過去の網羅性も保証されない
    → **前向き蓄積が本筋**（フェーズI）。過去分の後ろ向き検証は取れる範囲の副次的なものと割り切る。
- **フォールバック（有料）**: JPX公式 TDnet API（基本料7万円/月＋従量、過去5年取得可）。
  無料APIが不安定で検証に支障が出た場合の選択肢として§7に記録するのみ。**本書では使わない。**
- 対象は既存223銘柄のみ（v4§6と同じユニバース制約。拡張しない）。

### LLM: Claude API（`claude-api` skill 準拠。詳細はそのドキュメントに従う）
- **分類は高頻度・低難度**なので **`claude-haiku-4-5`（$1/$5 per MTok）** を主に使う。
  日次の開示件数は223銘柄でも1日あたり数件〜数十件程度で、コストは1日あたり数円オーダー。
- **要約や曖昧な判断が要る箇所のみ** `claude-opus-4-8`（$5/$25）を使う。`claude-fable-5`
  （$10/$50・最上位）は本書の定常処理には不要（コスト過大）。使うなら明示的に理由を書くこと。
- **構造化出力を必須にする**: `output_config.format` に json_schema を渡して分類結果の型を保証
  （Haiku 4.5 / Opus 4.8 とも対応）。パースは `json.loads` 相当で厳密に。
- **コスト最適化**: 定常の日次バッチは遅延非依存なので **Batch API（50%割引）** を検討してよい。
  ただし初期実装は素朴な逐次呼び出しで足りる（件数が少ないため）。
- **APIキー**: 環境変数（`ANTHROPIC_API_KEY`）。既存の Discord/Slack Webhook と同じく
  CIシークレット/ローカル環境変数で注入。**キーをコードに書かない・履歴に残さない。**
- モデルIDは skill の表を正とする（`claude-haiku-4-5` / `claude-opus-4-8` / `claude-fable-5`）。
  日付サフィックスを付けない。

## §3 フェーズI: 開示データ取り込み基盤（新規・単体実行＋日次ジョブ）

### 目的
223銘柄の適時開示を毎日取得し、Claudeで分類して**履歴として貯め始める**。この段階では
売買判断も表示変更もしない（貯めることが仕事）。**前向き蓄積の開始点。**

### I-1. 取得: `disclosure_fetch.mjs`（新規）
- Yanoshin API から「当日（または指定日）の223銘柄の開示」を取得し、
  `signals/disclosures.jsonl`（プルーン正規表現 state_*/signals_* に非該当）に追記する。
- 1レコード = `{date, code, title, docUrl, retrievedAt}`（本文は保存せず、分類結果を保存する方針。
  ただし分類に必要な範囲でタイトル＋PDF/HTMLの短い抜粋は一時取得してよい）。
- 冪等性: 同一 `date|code|title` は重複追記しない（Yanoshin再取得で二重計上を防ぐ）。
- 失敗時（APIダウン・タイムアウト）は**握りつぶさずログに残し、当日分をスキップして継続**
  （日次ジョブ全体を落とさない。欠損日はそのまま欠損として記録する）。

### I-2. 分類: `disclosure_classify.mjs`（新規・Claude API）
- 未分類の開示レコードを Claude（`claude-haiku-4-5`・構造化出力）に渡し、以下を**固定スキーマ**で得る:
  - `kind`: enum（`earnings`=決算 / `guidance_up`=上方修正 / `guidance_down`=下方修正 /
    `buyback`=自社株買い / `dividend`=配当 / `ma`=M&A・資本提携 / `other`=その他）
  - `surprise`: enum（`positive` / `negative` / `neutral`）— 開示内容が一般に好感/嫌気/中立か
  - `confidence`: number 0..1
  - `oneLine`: 20〜40字の日本語要約（表示用）
- **分類は監査可能に**: 入力（タイトル＋抜粋）・出力・モデルID・日時を残す。人手サンプル検証（§4）で
  精度を測るため。非決定性があるので**同一入力の再分類はしない**（初回結果を確定として保存）。
- 結果は `disclosures.jsonl` の各レコードに `cls:{kind,surprise,confidence,oneLine,model}` として付与。
- **未来を見ない**: 分類は開示当日の情報のみ（将来リターンを見せない）。当然だが最重要。

### I-3. 日次ジョブへの組み込み
- `screen_daily.mjs` の前段（またはCIの別ステップ）で `disclosure_fetch → disclosure_classify` を実行。
- **この段階では screen_daily の出力（レポート/通知）は一切変えない。** 蓄積のみ。
- APIキー未設定なら分類をスキップして取得だけ行う（キー無しでもジョブが壊れないこと）。

### 受け入れ確認
```
node disclosure_fetch.mjs --date <YYYY-MM-DD>   # 当日開示が disclosures.jsonl に入る・冪等
node disclosure_classify.mjs                    # 未分類レコードに cls が付く・構造化出力が型どおり
# → 数日運用して件数と分類分布をログ確認。screen_daily の既存出力が不変であることを確認。
# → 本書§7に「蓄積開始日・初期件数・分類分布」を記録してコミット。
```

## §4 フェーズJ: 検証（PEAD仮説の採点・データが貯まってから）

### 目的
「AIが分類した決算サプライズの方向に、翌日から市場平均を超えて動くか」を、既存の
成長窓リプレイ土台（v4の momentum_check.mjs と同じ物差し）で採点する。

### J-0. データ量の門番（事前固定）
- サプライズ付き開示（`kind=earnings` かつ `surprise∈{positive,negative}`）が **n≥200 件**
  貯まるまでフェーズJは実施しない。それ未満なら「データ蓄積中」と§7に記録して待つ。
- 無料APIで過去分が取れる場合は後ろ向きにも数えてよいが、**前向き蓄積分を主**とする
  （過去分は分類の当日性が担保しにくいため。取れなければ前向きのみで n を満たすまで待つ）。

### J-1. 計測: `disclosure_check.mjs`（新規・単体実行専用・CI外）
- v4 §2 の執行規約を**そのまま踏襲**: シグナル（開示）当日の翌営業日終値（t+1）で執行、
  **往復コスト0.2%控除後**で判定。当日終値執行は「理論上限」ラベルの参考値のみ。
- イベント: `earnings×positive`（買い方向）と `earnings×negative`（売り方向・符号反転）。
- 計測: t+1エントリーからの 5/10/20日 edge（対市場・classifyRegimes でレジーム別内訳）、
  勝率、**中央値**（分布の歪みを必ず見る。v4と同じ）。`confidence` の高低別内訳も出す
  （高信頼の分類だけならエッジがあるか、という切り分け）。
- 出力: テーブル＋ `signals/disclosure_stats.json`。

### J-2. 分類品質の検証（AIを信じる前に）
- **PEADの前に、分類そのものが正しいかを人手で確認する。** `earnings×surprise` の判定を
  ランダム30件サンプルし、実際の開示（上方/下方/中立）と突き合わせて**一致率**を出す。
- 一致率が **80%未満**なら、PEADの数字は分類ノイズに汚染されているとみなし、
  プロンプト改善（別コミット）を先に行う。§7に一致率を記録。

### 判断基準（事前固定・結果を見てから変えない）
- 「採用」= 分類一致率 ≥ 80% **かつ** コスト控除後 edge(10日) ≥ **+1.0%**（v2以来の物差し）
  **かつ** n ≥ 200 **かつ** edge(20日) > 0。
- 「現行維持の棄却」ではない: これは新規レイヤーなので、**採用ゼロなら情報レイヤーは
  「表示のみ（観察装置）」に留め、売買根拠としては出さない**（フェーズKの形が変わる）。

### 受け入れ確認
```
node disclosure_check.mjs
# → 完走・disclosure_stats.json 生成・分類一致率とedgeを表示
# → 結果と判定を本書§7に追記してコミット
```

## §6 制約と正直な注意（結果を読むときに必ず思い出すこと）

- **開示は公開情報＝即織り込み。** 適時開示は出た瞬間に誰でも見られ、価格に反映される。
  「AIが読んで気づく」時点で波の先頭は過ぎている。だから t+1執行・コスト控除後で判定する。
  PEADが効くとすれば「情報の完全な織り込みに時間がかかる」からであって、速報性ではない。
- **無料APIの脆弱性。** Yanoshin は個人運営で突然停止しうる。利用規約・レート制限を尊重し、
  過度な取得をしない。停止に備え、取得済みデータは追記型で確実に永続化する。商用/重要用途なら
  有料JPX APIだが本書では非スコープ。
- **LLM分類はコスト・幻覚・非決定性を持つ。** だから (a) 分類は監査可能に残す、(b) 人手サンプルで
  一致率を測る（J-2）、(c) 同一入力を再分類しない、(d) confidence を持たせて低信頼を割り引く。
- **前向き蓄積が本筋＝結果は数ヶ月先。** n≥200 のサプライズ開示が貯まるまで待つ。急がない
  （v2〜v4と同じく、時間をかけて事実を確定させることがこのプロジェクトの規律）。
- **ユニバースは大型223銘柄。** PEADは小型・情報の織り込みが遅い銘柄でより強いとされる。
  ここで駄目でも「大型では効かない」までしか言えない（v4§6と同じ非対称）。
- **これはツールの位置づけを変えない。** 予測ではなく整理・観察・記録（v2§7）。AIレイヤーが
  採用に至らなくても、それは「情報でも大型株の短期売買に単独エッジは無い」という正当な発見。

## §8 非スコープ（やらないこと）

- 自動売買・注文執行・ポジションサイズ指示（表示と記録まで）
- 有料データAPI（JPX TDnet API 等）の導入（無料で回す。フォールバック候補として記録のみ）
- 小型株・テーマ株など新ユニバースへの拡張（まず既存223銘柄。v4§8と同じ）
- リアルタイム/場中対応（日次バッチのまま）
- SNS・ニュースサイトのスクレイピング（適時開示という一次情報に限定。ノイズと規約リスクを避ける）
- LLMによる売買推奨文の生成（分類と要約まで。「買え/売れ」は書かせない）
- プロンプトの自動最適化・分類器のファインチューニング（過学習防止。プロンプトは手で改善）
- App.jsx / チャートUIの変更

## §9 実装セッションの進め方

1. 本書全体 → 旧設計書§2（不変条件）→ v2〜v4§7 → `claude-api` skill（モデルID・構造化出力・
   Batch API・エラー処理）→ `git log --oneline -10` の順で読む。
2. フェーズI（disclosure_fetch + disclosure_classify）を実装し、数日運用して蓄積を開始。
   件数・分類分布・既存出力の不変を§7に記録してコミット。
3. **n≥200 が貯まってからフェーズJ**（disclosure_check）。分類一致率→PEAD edge の順で採点し、
   §7に判定を書く。基準未達なら採用ゼロと明記。
4. **§7の判定を書いてからフェーズK**（K-a/K-b の該当する方）。着手前に 3bc5b56 を main にマージ。
   受け入れ確認（run_tests / screen_daily 2回 / LINE文字数）を通してコミット、push。
5. APIキー・外部API依存が入るので、キー無し/API停止時にジョブが壊れないことを各フェーズで確認。
