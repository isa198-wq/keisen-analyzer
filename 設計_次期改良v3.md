## §1 目的と評価軸（v2との決定的な違い）

v2の結論: 5年全域で、どの銘柄シグナルも単独で市場平均に勝てない（最良+0.23% < 基準+1.0%）。
ユーザーの方針はこれを受けて「**市場に乗る（銘柄選別に期待しない）＋ 大きな下落だけ避ける**」。
本書はその「避ける」側＝**退出警告**の検証と表示を設計する。

**評価軸が変わる。** v2は「市場平均に勝つか」だったが、v3は:

1. **最大ドローダウン(maxDD)をどれだけ削減できたか**（保険の効き）
2. **総リターンをどれだけ諦めたか**（保険料。ダマシで途中下車するコスト）
3. **ダマシの回数**（年に何回逃げさせられるか＝運用ストレス)

「天井の予測」はしない。やるのは**悪化の早期確認**（トレンドフォロー型の退出）であり、
上昇の最後の数%を諦める代わりに下落の大部分を避けられたかを測る。

対象は**このツールが見ている223銘柄の等ウェイト指数**（`buildMarketIndex`）。
日経平均そのものではない点は表示にも明記する（§5）。

## §2 使う部品とデータ（車輪の再発明禁止）

- `evaluate.mjs`: `buildMarketIndex(groups)` → `[{date, value, sma25, slope}]`
  （等ウェイト指数・SMA25・SMA25の5日差）。`classifyRegimes(groups)`
  （up = 指数>SMA25かつ傾き>0 / down = 逆 / range = それ以外）。`buildByCode` / `buildDateGrid`。
- `src/analysis.generated.mjs`: `analyze(buildSeries(tfSeries(bars.slice(0,i+1),"D")),"日")`
  の成長窓リプレイ（factor_check / combo_check と同一方式。1100日×223銘柄で実測41秒）。
- データ: `screening_data.csv`（5年・223銘柄）が主。`market_data.csv` のVIXは**2年分しかない**
  ため参考計測のみ（採否判定に使わない）。

## §3 フェーズE: 退出規則の成績計測（exit_check.mjs 新規・単体実行専用・CI外）

### E-1. 「逃げるべきだった局面」の定義（決定的アルゴリズム）

等ウェイト指数の**ドローダウン・エピソード**を列挙する:

1. `idx = buildMarketIndex(groups)` を先頭から走査し、ランニング最高値を更新したら
   そこまでの区間を締める（天井日=直近最高値の日、底日=区間内最安値の日）。
2. (天井値−底値)/天井値 ≥ **10%** の区間を「主要下落局面」とする（採否判定の対象）。
   ≥7%の区間も「参考局面」として出力する（判定には使わない）。
3. データ末尾で回復未了の区間も、その時点の深さで判定に含める。
4. 出力: 天井日 / 底日 / 深さ% / 所要営業日。**5年でおそらく3〜5件**のはず。
   0件または10件超なら定義かデータを疑い、§7に記録して停止する。

### E-2. 候補規則（手で列挙・閾値は事前固定・グリッド探索禁止)

各規則は「退出条件」と「復帰条件」のペア。**当日終値で判定し、翌日のリターンから反映**
（未来参照防止）。ヒステリシス（退出と復帰の閾値を分ける）でバタつきを抑える。

| key | 退出条件 | 復帰条件 | 閾値の根拠 |
|---|---|---|---|
| `regimeDown` | classifyRegimes が down | up になる（range は現状維持） | 既存分類器そのまま＝ベースライン |
| `maBreak3` | 指数がSMA25を**3日連続**で下回る | 3日連続で上回る | 3日=一晩のダマシ除去。パラメータ最少 |
| `breadth50` | トレンド因子がマイナスの銘柄が**過半数**(≥50%) | ≤35%に低下 | 「過半数が下向き」は閾値に自然な意味がある |
| `newLows10` | 60日終値安値を更新した銘柄比率の5日平均 ≥10% | ≤3% | 1割が新安値=指数が高くても内部崩壊 |
| `ddStop5` | 指数が250日高値から**5%**下落 | 高値から2%以内に回復 | 5%=指数の通常ノイズの上限程度、250日=1年 |
| `vixRef` | VIX ≥ 25 | < 20 | **参考のみ（2年分・採否対象外）** |

- `breadth50` のみ analyze() のリプレイが必要（他は終値だけで計算可）。リプレイは1回で
  全銘柄×全日の「トレンド因子の符号」「60日安値更新」を先に配列化し、規則判定は後段で行う。
- ウォームアップ: 指数の先頭260営業日は判定対象外（250日高値とSMA25の成立待ち）。
- **閾値の変種を後から試して採否を変えることは禁止。** 変えたくなったら§7に理由を書き、
  次回の検証で正式候補として追加する（v2と同じ規律）。

### E-3. 計測指標（規則ごと）

**A. 局面捕捉（E-1の主要局面それぞれについて）**
- 点灯日（天井日より後・底日以前の最初の退出シグナル。無ければ「捕捉失敗」）
- 天井からの遅れ（営業日）
- **capture% = (点灯日の指数 − 底値) ÷ (天井値 − 底値)** ＝下落のうち回避できた割合

**B. 5年 in/out シミュレーション（等ウェイト指数に対して。退出中は現金0%）**
- 総リターン / maxDD / ラウンドトリップ回数（退出→復帰=1回）/ 市場滞在率
- 同じ値を buy&hold と並記
- ダマシコスト: 復帰時の指数が退出時より**高かった**ラウンドトリップの回数と平均損失%

### E-4. 出力

- テーブル表示（規則×上記指標、capture%平均の降順）＋ E-1の局面一覧
- `signals/exit_stats.json` に保存（プルーン正規表現 state_*/signals_* に一致しない名前でOK）
- 使い方: `node exit_check.mjs`（既定=全期間。`--days N` は不要、常に全期間でよい）

### E-5. 判断基準（事前固定。結果を見てから変えない）

「採用」= 以下を**全て**満たす:

1. 主要下落局面（≥10%）の**全件**で点灯し、capture%の平均 ≥ **50%** かつ 最悪の局面でも ≥ **30%**
2. 5年シミュレーションの maxDD ≤ buy&hold の maxDD × **0.7**（3割以上の削減）
3. 5年総リターン ≥ buy&hold × **0.7**（保険料の上限）
4. ラウンドトリップ ≤ **年4回**（5年で20回）

- 複数採用 → **maxDD削減が最大の1つを主警告**、残りは補助（点灯数として表示）。
- **採用ゼロもあり得る正当な結果**。その場合フェーズFは実施せず、既存のレジーム行のみ継続
  （§7に記録）。無理に閾値をいじって通さないこと。

### 受け入れ確認

```
# → 結果表とE-5の判定を本書§7に追記してコミット
```

## §4 フェーズEの実装メモ

- combo_check.mjs の構造（CSV読み込み→成長窓リプレイ→集計→テーブル＋JSON）を踏襲する。
- 規則の定義は `exit_defs.mjs` に分離（combo_defs.mjs と同じ思想。フェーズFで screen_daily と
  共有するため）。各規則は「日次の集計値（指数・SMA25・breadth比率・新安値比率・VIX）を受け取り
  退出/復帰を返す純関数」とし、状態機械（現在in/outか）は呼び出し側が持つ。
- シミュレーションの警告: シグナル判定は当日終値、ポジション反映は翌日リターンから。
  ここを間違えると1日先読みになり結果が過大評価される（テストで最重要の確認点）。

## §5 フェーズF: 退出警告の表示（Eで採用が出た場合のみ・表示のみ・シグナル定義は不変）

1. **screen_daily.mjs**: 既存の analyze() 全銘柄パスの結果を再利用して当日の breadth・新安値
   比率を集計（再計算しない）。`exit_defs.mjs` の採用規則で当日状態を判定。
2. **表示**: レポートの regimeLine の直後に1行。点灯中のみ
   `⚠️ 退出警告: <規則ラベル> 点灯中（YYYY-MM-DD から / 対象はツール内223銘柄の等ウェイト指数）`。
   消灯中は `退出警告: なし` を市況行に併記（毎日見るものなので「無い」ことも明示）。
   LINE/Webhookにも点灯中のみ1行追加。**既知のLINE 4900字超過問題があるため、追加後の
   想定文字数を必ず確認**（超過が悪化するなら警告行を本文の先頭に置き、切り詰めの影響を受けない
   位置にする）。
3. **history.jsonl**: 点灯中のみ `exit:{"<ruleKey>":1}` を当日エントリに付与（答え合わせの
   継続測定用。旧スキーマ互換: `e.exit || {}`。消灯日はキー省略）。evaluate() の CATS は変更しない。
4. **状態の持ち方**: in/out 状態は history.jsonl の前日エントリの `exit` から復元する
   （state_*.json は作らない。復帰条件の判定に必要な「点灯開始日」は exit の値を
   `{"<ruleKey>":"YYYY-MM-DD"}` として開始日で持てば1フィールドで足りる）。

### 受け入れ確認
```
node run_tests.mjs                       # 全PASS（シグナル定義は不変）
SIGNALS=strong node screen_daily.mjs     # 2回連続: 警告行表示・history冪等・LINE文字数確認
```

## §6 制約と正直な注意（結果を読むときに必ず思い出すこと）

- **n が小さい。** 主要下落局面は5年で3〜5件しかない。E-5を全て満たしても「数回の事例全部で
  機能した」以上のことは言えない。だから規則は単純・閾値は少数・事前固定に限定した。
- この5年は上昇相場が支配的（上昇レジーム約75%）。退出規則の「保険料」（基準3）は
  上昇相場では重く見積もられる＝**通れば辛口の検証を通ったことになる**。逆に下落の捕捉（基準1)
  は事例が少なく甘くなりがち＝**通っても過信しない**。
- 採用後も答え合わせを継続する（§5-3の記録が材料）。半年〜1年ごとに exit_check.mjs を再実行し、
  基準を割ったら §7 に記録して警告を解除する（格上げと同じ規律で格下げする）。

## §7 判断記録（実装セッションが追記していく欄）

> フェーズEの実行結果と採否判断をここに表で追記する。**基準は§3(E-5)に事前固定した
> ものを使い、結果を見てから変えない。** 変えたくなったら理由を書いて明示的に変更する。

- 2026-07-05: 本書作成（Fable）。背景はv2§7の総合判断（銘柄選別のエッジ無し→市場に乗り
  大きな下落だけ避ける方針への転換）。

### 2026-07-05 フェーズE実行結果（`node exit_check.mjs`、全1,222営業日×223銘柄、21秒）

**E-1 局面検出**: 主要局面(≥10%)は**4件**（想定3〜5件どおり）。
2021-09-14→2022-03-09(13.8%・117日) / 2024-07-17→2024-08-05(22.8%・13日) /
2025-03-26→2025-04-07(18.0%・8日) / 2026-02-27→2026-03-23(12.7%・15日)。
参考(7〜10%)が3件。全期間buy&hold: 2022-07-28以降だけで**+153.0%**（この5年は極端な上昇相場）。

**E-3 規則別成績**（非参考5規則。readyFromは規則ごとのウォームアップ終了日）:

| 規則 | 局面捕捉(全件/平均/最悪) | リターン(B&H比) | maxDD(B&H比) | 年間往復 | 採用可否 |
|---|---|---|---|---|---|
| maBreak3(SMA25×3日) | ×/64.4%/50.9% | +74.0%(48%) | 19.3%(85%) | 5.0回 | 見送り |
| ddStop5(250日高値-5%) | ×/59.6%/35.9% | +75.3%(49%) | 10.8%(47%) | 2.1回 | 見送り |
| regimeDown | ×/50.4%/47.0% | +68.8%(45%) | 18.2%(80%) | 3.7回 | 見送り |
| newLows10(新安値1割) | ×/42.3%/**0.0%** | +105.8%(69%) | 13.5%(59%) | 3.1回 | 見送り |
| breadth50(過半数下向き) | ×/-（一度も点灯せず） | +80.2%(52%) | 22.8%(100%) | 1.6回 | 見送り |
| vixRef(参考) | ×/35.7%/0.0% | +25.4%(41%) | 22.8%(100%) | 3.6回 | (参考・対象外) |

**判断（§3 E-5基準を適用）: 採用ゼロ。フェーズFは実施しない。**

構造的な理由が2つ重なっている:

1. **局面1(2021-09-14〜)が全規則のウォームアップ(260営業日=2022-07-28まで)より前で終わっており、
   定義上どの規則も点灯不可能**（データが5年しかなく、最初の下落が1年目に来たため）。
   これにより「全件点灯」を満たす規則はゼロになる。規則の性能不足ではなくデータ期間の制約。
2. **測定可能な3局面に限っても、リターン維持基準(≥70%)を割る。** この5年はbuy&holdが+153%という
   極端な上昇相場で、どの規則も一時退出のたびにその上昇を取り逃がし、最良のnewLows10でも69%
   （僅かに届かず）。§6で事前に警告したとおり「上昇相場では保険料が重く見える」が実際に効いた。
   maxDD削減が明確に効いたのはddStop5(53%減)とnewLows10(41%減)のみで、他は不十分。
   breadth50は一度も点灯せず（トレンド因子の過半数反転は短期急落に対して反応が遅すぎる。
   v2で確認した「トレンド因子は全期間ではエッジが小さい」こととも整合）。

**総合判断**: 現状のデータ・規則では採用に値するものはない。個々の候補で見どころはある
（ddStop5のDD削減効率、newLows10の捕捉率）が、事前固定基準を単独で満たすものはゼロ。
**フェーズFは実施しない（§9のとおり）。** 今後データが増え(局面1がウォームアップを抜ける、
あるいは下落相場が追加される)、再計測で基準を満たせば改めて採用を検討する。

## §8 非スコープ（やらないこと）

- 自動売買・注文執行・ポジションサイズの指示（表示と記録まで）
- 天井の予測・逆張りの底当て（やるのは悪化のトレンドフォロー型確認のみ）

- 「買い増し時」「押し目」の判定（本書は退出のみ。入り直しは復帰条件の機械的判定まで）
- market_data.csv の期間拡充や新規データソース追加（VIXは参考計測に留める）
- App.jsx / チャートUIの変更

## §9 実装セッションの進め方

1. 本書全体 → 旧設計書§2（不変条件）→ v2§7（判断記録）→ `git log --oneline -10` の順で読む。
2. フェーズE（exit_defs.mjs + exit_check.mjs）を実装して実行。結果を§7に追記してコミット。
3. **E-5の判定を§7に書いてから**フェーズFに着手（採用ゼロならFはスキップし、その旨を§7へ）。
4. F実施時は受け入れ確認（run_tests / screen_daily 2回 / LINE文字数）を通してコミット、push。
