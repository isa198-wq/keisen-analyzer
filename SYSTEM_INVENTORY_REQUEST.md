# 依頼:イナゴ・トレード盤エコシステム 現状棚卸し

## 目的

いろいろ作りすぎて全体像を見失っているため、**ローカルに存在するファイル群を実際に読んで、今何が・どこまで動いていて・何が未接続かを1枚のマップにまとめる**。新しい実装・修正は一切しない。今回は調査とドキュメント化のみ。

## Step 1:フォルダ全体をスキャン

対象:`C:\Users\isa19\OneDrive\ドキュメント\keisen-analyzer`

以下を実行し、全体像を把握する:

```powershell
Get-ChildItem "C:\Users\isa19\OneDrive\ドキュメント\keisen-analyzer" -Recurse -File | 
  Select-Object FullName, Length, LastWriteTime | Sort-Object LastWriteTime -Descending
```

このフォルダ以外にも関連ファイルがないか、念のため以下も確認する:

```powershell
Get-ChildItem "C:\Users\isa19\OneDrive\ドキュメント" -Directory
```

（keisen-analyzer以外に、それらしきフォルダ(discipline関連、trading関連等)がないか確認するため。あれば同様にスキャン対象へ追加してユーザーに報告する）

## Step 2:各ファイルの中身を実際に読んで判定する

**ファイル名や最終更新日だけで判断しない。**特に同じ役割に見える複数ファイル(例:`inago_daily.html` と `inago_offline.html`)は、実際に開いて中身を比較し、「どちらが本物で、どちらが放棄版か」を判定する。判断基準の例:

- HTMLファイル:CDN読み込み(`<script src="https://...">`)があるか→あれば実行環境的に開けない可能性が高い(過去に判明した既知の問題)
- `window.INAGO_DATA` を参照しているか、`data.js` は実在するか、中身は合成データ(乱数生成コード)か実データ(実際の証券コード+日付付きcandles)か
- Pythonスクリプト:import しているモジュール、必要な環境変数(例:`XAI_API_KEY`)、最後に動いた形跡(ログファイル等があれば)
- PowerShellスクリプト:タスクスケジューラ登録の有無・スケジュール内容
- 各ファイルの冒頭コメントに設計意図が書かれていることが多いので必ず読む

## Step 3:分かっていること(前提情報)との突合

以下は既に判明している情報。矛盾がないか確認しつつマップを作る:

- 本体は `inago_offline.html`(CDN不使用・自前SVG/JS版)。`inago_daily.html`(React/CDN版)は放棄版で、監査時点では構文エラーで開けない状態だった(閉じタグ欠落)
- `data.js`(`window.INAGO_DATA`)は罫線アナライザーとの橋渡し用に設計されたが、**未接続**。現状は合成デモデータで動作
- `grok_name.py`:xAI APIでクラスタ命名。`XAI_API_KEY`環境変数が必要
- `detect_clusters.mjs`:HTMLのスコアリングロジックをNode単体で再現し、クラスタ検出をヘッドレス実行
- `run_inago.ps1`:上記の日次自動実行。`-Register`フラグでタスクスケジューラ(毎日15:45)に登録
- `discipline.html` + 夜間バッチPythonスクリプト:逆指値による機械的な手仕舞い、ルール遵守率・ペイオフレシオ追跡(このフォルダにあるか要確認)

## Step 4:成果物 — 棚卸しマップ

`SYSTEM_MAP.md` として以下の構成でまとめる:

### 4-1. ファイル一覧表

| ファイル名 | 場所 | 役割 | 状態(稼働中/放棄/未接続/不明) | 根拠 |
|---|---|---|---|---|

### 4-2. データフロー図(テキストベースで可)

罫線アナライザー → (未接続の矢印) → data.js → inago_offline.html → 各タブ、のように、実際に繋がっている部分と繋がっていない部分を視覚的に区別する

### 4-3. 「今すぐ使える状態のもの」リスト

今日開いてそのまま使えるものだけを列挙

### 4-4. 「動くが未接続/未検証」リスト

コードは存在するが、実データ未接続・実行未確認のもの(例:検証タブは合成データのまま)

### 4-5. 「重複・混乱の原因になっているもの」リスト

同じ役割の複数バージョンが存在するファイル。どれを正、どれを削除・アーカイブ候補とすべきかの提案付き

### 4-6. オンライン側(GAS/Notion/LINE)についての注記

ローカルから確認できない部分は「要ユーザー確認」として空欄のまま残し、後述の質問リストに回す

## Step 5:オンライン側の確認リストを作る

Claude Codeからは見えないGAS/Notion/LINE側について、ユーザーに後で答えてもらうための**チェックリスト**を `ONLINE_SYSTEMS_CHECKLIST.md` として作る。次の粒度で、埋めるだけの形にする:

- [ ] 生活バランスシステム(GAS+LINE+Notion日記):GASプロジェクト名/URL、最終更新、動作確認済みか
- [ ] NativeCampレビューパイプライン(Notion MCP + GAS + LINE):3つのNotion DB名、稼働確認日
- [ ] 週次市場レポート自動化(Claude Code scheduled job → Notion → GAS → LINE):最終実行日、直近正常に届いたか
- [ ] 気になりメモ(LINE→Claude API→Notion+日記):Makeシナリオ名、稼働状況
- [ ] フォローアップトラッカー(SQLite試作):ファイルの場所、プロトタイプ止まりか稼働中か

## 完了条件

1. `SYSTEM_MAP.md`(ローカル資産の実態、根拠付き)
2. `ONLINE_SYSTEMS_CHECKLIST.md`(オンライン資産の確認待ちリスト)
3. 上記2ファイルの内容を口頭で3行要約(「今動いているのはこれだけ」「一番の詰まりポイントはここ」「次に手を付けるべきはここ」)

## 注意事項

- 推測で埋めない。ファイルを開いて確認できたことだけを書く。確認できなければ「不明」と明記する
- 新規実装・修正・リファクタは一切しない。今回は現状把握のみ
- ファイル全文の転記は不要。役割と状態が分かる程度の要約でよい
