### 事実1: 三尊・逆三尊の完成イベントは、単独では方向どおりのエッジがない

5年・223銘柄・シグナル1,229件のバックテスト（`node backtest.mjs`、2026-07-04実行）:

- 三尊完成後はむしろ市場より上がる。逆三尊は市場に負ける。20日でも同傾向。
- 週足◎の複合でも改善しない（旧§1-事実3を大サンプルで追認）。
- **ただし注意**: この5年は上昇相場が支配的（等ウェイト指数で上昇レジームが約75%）。
  「下落局面では三尊が効く」可能性はレジーム条件付き検証（フェーズB）で判定する。

### 事実2: ファクターの有効性に明確な序列がある

過去120営業日×223銘柄=26,760セルのリプレイ（`node factor_check.mjs --days 120`、
結果は `signals/factor_stats.json`）。edge = 方向調整済みの対市場優位（10日後）:

| ファクター | n | edge(10日) | レジーム別（上昇/下落/もみ合い） |
|---|---|---|---|
| トレンド | 18,530 | **+0.99%** | +1.3% / +0.6% / +0.1% |
| クロス | 3,973 | **+0.84%** | +1.1% / +1.1% / -0.4% |
| 三尊(factor) | 9,584 | +0.36% | +0.5% / -0.1% / +0.2% |
| 逆三尊(factor) | 10,173 | +0.23% | +0.2% / -0.0% / +0.3% |

| BB | 3,520 | -0.57% | -0.8% / +2.5%(n=113) / -0.1% |
| RSI | 3,102 | **-1.04%** | -1.1% / +1.3%(n=51) / -0.5% |

- トレンド・クロスは全レジームで頑健にプラス。
- RSI・BBはマイナス（逆指標疑い）。ただし**計測窓が120日=ほぼ上昇相場のみ**で、

- MACDは下落時に大きく負ける。

### 結論（本書の方針）

1. **主役の交代候補**: 「三尊/逆三尊の完成」中心の通知から、「トレンド×クロス系の
   複合条件」中心への格付け変更を、**追加検証で裏取りしてから**行う。
2. **段階主義**: いきなり重み(s値)を変えない。A(検証窓拡大)→B(複合条件計測)→
   C(表示の格付け変更)→D(重み格下げ・条件付き)の順で、各段階の判断基準を事前に固定する
   （後出しの基準変更は過学習と同じ）。
3. **情報は隠さない**: パターン検出・全シグナルの表示と履歴蓄積は続ける
   （検証を止めないため。旧§2不変条件のとおり）。

## §3 フェーズA: 検証窓の拡大（RSI/BB/MACD逆指標疑いの裏取り）

### 目的
事実2の計測は直近120日（上昇相場に偏る）。「RSI/BBの負edgeが全期間・全レジームでも
成り立つか」を確定させ、フェーズDの判断材料にする。**コードの新規開発はほぼ無い。**

### 仕様
1. `factor_check.mjs --days 1100` を実行（5年全域リプレイ。--days は実装済み）。
   実行時間の目安: 120日で約1秒だったため、1100日でも数十秒〜数分。
2. 出力JSONの保存名が上書きにならないよう、`--out <name>` オプションを追加
   （既定 `factor_stats.json`。全期間実行時は `--out factor_stats_full.json` を使う）。
   プルーン正規表現（state_* / signals_*）に一致しない名前であること（両方とも一致しない）。
3. 結果を本書の§7（判断記録）に**表で追記**する（レジーム別 n が数百以上になるはず）。

### 判断基準（事前固定）
- 「見直し確定」= 全期間edge(10日) < -0.3% **かつ** 下落レジームedge(10日) < 0。
- 「保留」= 全期間は負だが下落レジームで明確に正（例: RSI逆張りは下落時のみ機能）。
  この場合フェーズDでは重みを変えず、レジーム条件付き表示（フェーズC）の材料にする。

### 受け入れ確認
```

# → 完走・signals/factor_stats_full.json 生成・下落レジームの n が数百以上
```

## §4 フェーズB: 複合条件の成績計測（combo_check.mjs 新規）

### 目的
「どの組み合わせなら通知に値するか」をデータで決める。候補は手で列挙した固定セット
（自動探索はしない＝過学習防止。旧§10の不変方針）。

### 仕様
`combo_check.mjs`（新規・単体実行専用・CIに入れない）:
- リプレイ方式はフェーズ4/5と同じ（生バー成長窓: `buildSeries(tfSeries(bars.slice(0,i+1),"D"))`
  → analyze、フォワードリターンは生バー、ベースラインは同日全銘柄平均、
  classifyRegimes でレジーム別内訳）。`--days N` 既定 1100。
- **判定材料は analyze() の戻り値のみ**（factors の k と s、vIdx、pattern）。App.jsx は変更しない。
- 固定コンボセット（この定義をそのままコードに写す。追加・変更はコミットメッセージに理由必須）:
  1. `strongBuy`: vIdx===4（現行の通知条件＝ベースライン比較用）
  2. `strongSell`: vIdx===0（同上）
  3. `trendCross`: factors に トレンド(s>0) と クロス(s>0) が両方ある（買い方向）
  4. `trendCrossBear`: トレンド(s<0) と クロス(s<0)（売り方向）
  5. `trendCrossUp`: 3 かつ その日のレジームが up（レジームは判定に使ってよい。
     等ウェイト指数は当日終値までで決定的に計算できるため未来参照にならない）
  6. `invTrend`: 逆三尊 confirmed（pattern.kind==="inverse" && broke）かつ トレンド(s>0)
  7. `topTrendBear`: 三尊 confirmed かつ トレンド(s<0)
  8. `strongBuyDown`: vIdx===4 かつ レジーム down（「下落時の買いは効かない」仮説の直接検証）
- 出力: コンボごとに n / 出現率 / edge(5/10/20日) / 勝率 / レジーム別edge(10日)。
  `signals/combo_stats.json` に保存。テーブルは edge(10日) 降順。
- 方向: 1,3,5,6 は買い（リターンそのまま）、2,4,7 は売り（符号反転して集計）。8は買い。

### 判断基準（事前固定）
- 「採用」= edge(10日) ≥ +1.0% かつ n ≥ 300 かつ 20日edgeも正。
- 「現行維持の棄却」= strongBuy/strongSell 単独が採用基準を満たさない場合、
  フェーズCで単独シグナルの表示格下げを行う根拠になる。

### 受け入れ確認
```
node combo_check.mjs --days 1100
# → 完走・combo_stats.json 生成・strongBuy の n が数千オーダー
# → 結果を本書§7に追記してコミット
```

## §6 フェーズD: 重みの格下げ（条件付き。A/Bが支持した場合のみ）

### 発動条件（全て満たすときだけ着手）
- フェーズAで RSI または BB が「見直し確定」（全期間かつ下落レジームでも負）。
- フェーズCの運用開始から2週間以上、⭐注目条件の日次成績が破綻していない。

### 仕様
- **反転はしない**（負edgeだから逆張りに使う、は過学習の典型）。やるのは**減衰のみ**:
  対象ファクターの s 値を半分（例: ±2→±1）にする。0化は次の再計測後まで保留。
- App.jsx の analyze() 内で対象 factors.push の s を直接書き換える（変更は最小差分）。
  **App.jsx を触るので必ず `node build_analysis.mjs` → `node run_tests.mjs`**。
  golden fixture が vIdx 変化で壊れる場合はテスト側の期待値を「理由付きで」更新する。
- 変更前後で `node backtest.mjs` と `node factor_check.mjs --days 1100` を再実行し、
  スコア分布の変化（買い/売り件数の増減）をコミットメッセージに記録する。
- vIdx の閾値（verdict の区切り）は変更しない。

### 受け入れ確認
変更前後の factor_check / backtest / screen_daily(2回) の比較値をコミットメッセージに記載。
